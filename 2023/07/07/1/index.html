发表于 2023-07-07 分类于 MySQL
本文字数： 2.1k 阅读时长 ≈ 2 分钟
本文介绍了数据库的设计，以及三大范式。三大范式是数据库设计的基本原则，它可以帮助我们设计出更好的数据库。我们需要权衡规范与性能，以提高开发效率。

设计一个项目的数据库

为何要设计数据库

在项目开发的过程中，我们需要存储数据，而数据库就是存储数据的地方。数据库的设计，可以让我们更好地存储数据，更好地管理数据，更好地使用数据。

👎Bad design:

  • 数据冗余，浪费存储空间

  • 数据不一致，修改数据时，需要修改多处

  • 程序性能低，查询数据时，需要多次查询

👍Good design:

  • 数据不冗余，节省存储空间

  • 数据一致，修改数据时，只需要修改一处

  • 方便开发

软件开发中，数据库设计是一个非常重要的环节，它关系到软件的性能、可扩展性、可维护性等。
🤔我们要怎么设计数据库呢？

  • 需求分析：分析项目的需求，确定需要存储哪些数据

  • 概念设计：将需求转换为数据库的概念模型

设计数据库的步骤(适用于个人博客)：

  1. 收集信息，分析需求

    • 用户表
    • 分类表
    • 文章表
    • 评论表
    • 友链表
    • 自定义表

  2. 确定实体，将需求落实到每个字段

    • 用户表：用户 ID、用户名、密码、邮箱、头像、创建时间、更新时间
    • 分类表：分类 ID、分类名称、创建时间、更新时间
    • 文章表：文章 ID、标题、内容、创建时间、更新时间、作者 ID
    • 评论表：评论 ID、评论内容、评论时间、评论者 ID、文章 ID
    • 友链表：友链 ID、友链名称、友链地址、创建时间、更新时间
    • 自定义表：系统信息、某个关键字、某个配置项

注意： 命名时不要使用驼峰命名法，而是使用下划线命名法，例如：user_id、user_name、create_time、update_time 等。因为在 MySQL 中，不区分大小写。如果使用驼峰命名法，那么在 SQL 语句中，就需要使用反引号来包裹字段名，例如：userId、userName、createTime、updateTime 等。这样会增加 SQL 语句的复杂度，不利于开发。

  1. 确定实体之间的关系

    • 用户表和文章表：一对多
    • 文章表和分类表：多对多
    • 文章表和评论表：一对多
    • 文章表和用户表：一对多
    • 友链表和用户表：一对多
    • 自定义表和用户表：一对多
    • 写作：user > article
    • 分类：article > category
    • 评论：article > comment
    • 作者：article > user
    • 友链：user > link
    • 自定义：user > config

三大范式

为什么要遵循范式？

  • 数据冗余

  • 更新异常

  • 删除异常 – 丢失有效信息

  • 插入异常 – 无法正常显示信息

参考：

  • 数据库设计的三大范式

  • 数据库设计的三大范式以上两个链接中的内容有些许不同，但是都是对三大范式的解释，可以参考。

第一范式 1NF

第一范式是指数据库表的每一列都是不可分割的基本数据项，同一列中不能有多个值，即实体中的某个属性不能有多个值或者不能有重复的属性。
例如，用户表中的用户名字段，不能存储多个用户名，也不能存储重复的用户名。

eg:
用户表：

idnameagegendercontact
1张三18男12345678901
2李四20男12345678902
3王五22男12345678903

第二范式 2NF

第二范式是在第一范式的基础上建立起来的，第二范式要求数据库表中的每个实例或行必须可以被唯一地区分。通常需要为表添加一个主键，以便能够唯一地区分表中的每一行。例如，用户表中的用户 ID 字段，就是一个主键，可以唯一地区分表中的每一行。

eg:
用户表：

idnameagegender
1张三18男
2李四20男
3王五22男

文章表：

idtitlecontentauthor_id
1标题1内容11
2标题2内容21
3标题3内容32

上面的文章表中，author_id 字段就是一个外键，它引用了用户表中的用户 ID 字段，可以唯一地区分用户表中的每一行。

第三范式 3NF

第三范式是在第二范式的基础上建立起来的（即同时满足第一范式和第二范式），第三范式要求数据库表中的每一列数据都和主键直接相关，而不能间接相关。例如，用户表中的用户名字段，和用户 ID 字段直接相关，而和用户的邮箱字段间接相关。

eg:
文章表：

idtitlecontentauthor_idcategory_id
1标题1内容111
2标题2内容212
3标题3内容321

分类表：

idnamecreate_timeupdate_time
1分类12021-07-07 21:38:042021-07-07 21:38:04
2分类22021-07-07 21:38:042021-07-07 21:38:04

为了满足第三范式，我们需要将文章表中的分类 ID 字段，拆分为分类表中的分类 ID 字段和分类名称字段。

规范与性能

关联查询的性能比较差，所以在设计数据库时，需要权衡规范与性能。以阿里公司的数据库设计规范为例，它要求数据库表中的每个字段都要有注释，但是不要求满足第三范式。参考：阿里巴巴Java开发手册（终极版）

  • 考虑商业化的需求与目标（如成本、效率、安全等），合理设计表结构，尽量减少关联查询，以提高性能。

  • 适当考虑冗余字段，以提高性能。（例如，用户表中的用户名字段，可以冗余到文章表中，这样就不需要关联查询用户表了）

  • 适当考虑 规范性 与 性能 的平衡，以提高开发效率。

  • 故意增加一些计算例 （从大数据量中查询少量数据，可以使用计算列，以提高性能）
