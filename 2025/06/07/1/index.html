SDG320 en7562ct ax3000 研究

发表于 2025-06-07 更新于 2025-06-27 本文字数： 8k 阅读时长 ≈ 29 分钟
SDG320 这个型号的路由器，拆机之后发现是 EN7562CT 方案的，使用了 MT7976DN 和 MT7916AN 的组合。二手网买的本想捡漏的，一看企业级，5天线，还以为是 MT7981。

路由器型号：AX3000

OEM 型号：SDG320

方案：EN7562CT+MT7976DN+MT7916AN

  • SoC: EN7562CT: ARM Cortex-A53 DUAL-CORE@900MHZ with 1 NPU

  • RAM: DDR3 512MB:K4B4G1646E-BYMA

  • SPI NAND: 256MB:HYF2GQ4UTACAE

这颗 SoC 的频率可能是 900MHZ、950MHZ 1GHZ，看厂家设置，好像还包含以一个 NPU，而且现在 ECONET 合并至 Airoha。

末尾的 CT 指的是支持 XGPON 的意思，但是这台设备是纯路由器。CT相比DT，少了一个 NPU。还有一个 CU 版本，集成 512MB 内存。

但是这台机器，我严重怀疑，他的 NPU 根本没用上，没有开启硬件加速，因为测试下来，比 MT7621AT@880Mhz（enable MediaTek HWNat） 还慢，但是看编译选项，HWNat又是开启的。

另外提一嘴，SDG320 应该是政企版本，还有一个型号是 SDG400T，使用双核 900MHz 的处理器 512MB（估计是一个东西对应不同采购，企业版本），SDG400L 使用双核 1GHz 的处理器256MB（应该是家用版本），还有个上海的 SDG440（内置 audit control，4网口4语音1光口，无线方面阉割），政企网关 SDG400（与SDG440 类似，运营商不同），都是一个系列的产品。还有某西部地区特制版本，型号不明，但是内置严格的审计控制。OT1800 系列应该也是使用达发芯片（可能是en7561），无线方案换成 AX1800。

分区解析

分区号分区名描述
mtd0bootloaderU-Boot 引导加载器
mtd1rescue恢复分区，包含整个系统镜像
mtd2reservearea保留区域，实际存储了 art 信息
mtd3romfileROM 文件，实际存储了一些补丁和 web 后台页面的文件
mtd4kernela内核 A 分区，包含主内核和驱动
mtd5rootfsa根文件系统 A 分区，包含系统文件和配置
mtd6kernelb内核 B 分区，备用内核
mtd7rootfsb根文件系统 B 分区，备用系统文件
mtd8data数据分区，jffs2
mtd9javaJava 虚拟机分区，实际为空
mtd10osgiOSGi 框架分区，实际为空

这个reservearea 分区，偏移量0x40000到0x60000，存储了 wifi（MT7916D_EEPROM） 校准信息；偏移量0x60100到0x604C0，存储了 mac 地址和序列号等信息，从0x60180开始，这里是 MAC 地址的信息，不过并非 16 进制数为 MAC 地址，而是转为 ASCII 后为 MAC 地址，该 mac 为标签 mac，同时为 lan mac，wan mac 是 lan mac 基础 + 2；从0x604A0开始长度64，该段数据作用未知；偏移量0xC0000到0xC2950，大小 0x2950，存储了一段信息，目前还不清楚具体用途。

从原厂的编译信息分析

/userfs/profile.cfg 有着一些编译信息。

这个分区信息是移动版本的，其他版本的是下面这样的

UBOOT 2014 版本 ATF2_1

这两个 SECURE_BOOT 有什么区别吗？应该上面这个是 ATF 的。

如果有高手，可以结合开源的 ATF 代码，尝试制作一个多功能 UBOOT，就像 uboot-mt798x 一样

这里只选取了最重要的差别，其他系统定义上的差别、驱动特性的差别没有给出。
因为没有注释，只能推测

第一行 TCSUPPORT_ETHPHY=5R_DIS 可能是指禁用了 5 个以太网物理接口（可能是为启用 en8850 交换机做准备），也可能是其他含义。

第二行 TCSUPPORT_7562_CD=y 可能是指启用了 en7562ct 的特性，或者单纯是一个标志位，代表该芯片来自 7562 家族。

第三行 TCSUPPORT_GCC10_3_0_GLIBC2_32_KERNEL5_4=y 指定了编译环境，使用 GCC 10.3.0、glibc 2.32 和 Linux 内核 5.4，符合来自下面提到的编译信息。

第四行 TCSUPPORT_SWITCH_8850=y 指定了使用 8850 交换机芯片。这个 8850 应该不是至 rtl8850，而是达发/MTK的 8850 交换机芯片，因为 rtl8850 是一个Wifi+蓝牙芯片，所以这个 8850 应该是达发的 8850 交换机芯片。

第五行 TCSUPPORT_BUSYBOXCONFIG_TC3262=y 指定了使用了 TC3262 的 BusyBox 配置。TC3262 是达发的一个 SoC 系列，可能是为了兼容性或特定功能。

TcBoot 是从 TrendChip 的 TC3262 衍生而来的极简引导加载程序。可能是为了支持 A/B 分区的相关特性，或是 NAND BBT/BMT 相关功能。

某些编译信息，来自固件里打包的某些文件，这应该是编译 Airoha SDK 的工具链。

没有源码，看个乐呵。

设备树构建猜想以及 SoC 组成猜测

以下是进行推测的过程，可以直接跳到末尾最新的部分。

当然，这些都是推测，不保准准确，没有看到 dieshot 时，谁也不能确定。

airoha: add support for Gemtek W1700K

这个提交展示了如何在带有 Secure Boot 的达发设备上引导系统。W1700K 是一个基于 an7581 的设备，使用了 Secure Boot 机制。

我感觉有戏，感觉可以为该设备提供 OpenWrt 支持。但是主线上的 an7523 只有验证样品的 DTS 文件，没有模板。

不过话又说会来了，这个 dts 貌似不太完整，比如 keys、led 的 GPIO 都没有，而且也没有分区信息，应该是藏在内核脚本里了，还是要ttl看启动信息。

又看了一下 OpenWrt 主线的 en7523.dtsi，给我整不会了，什么 gsw、eth、switch、gdm、sgmii 之类的定义都没有(我还看了Linux的达发芯片适配代码，也是一样的)，也就是说，就算使用 an7523 的 dts，做出来系统也就是能启动这个 SoC，没有交换机，网口都不能用，不过有 pcie，无线应该能启动。

而 an7581 的 dtsi 比较合适，相比之下，en7562ct要少几个 cpu，频率也低一些，但是网口部分 an7581 用到的 gdm1 和 gsw，而 en7562ct 用到的是 gdma 和 sgmii。

我不太明白 gdm 和 gdma，到底 gdm 是以太网 mac 模块，gdma 同样如此，后缀a是编号还是代表 dma 模块，这个命名太迷惑了，也就是说，目前 en7562ct 的 dts 还不完整，主要是以太网部分，如果能够完善这一部分，路由器产品应该就能适配 OpenWrt 了（当然如果是光猫设备的话，还缺少了 xpon、pon 之类的设备树） 见参考资料 [1]。

破案了，gdma 应该是一个 dma 设备，而 gdm 是一个以太网 mac 设备，frame_engine 集成了以太网相关的部分。

从设备树来看，有1个 pon_hsgmii（2.5Gbps），3个 sgmii（sgmii_p0、sgmii_p1、sgmii_u0，1Gbps），不清楚是否将 pon_hsgmii 作为 wan 口使用

但是在 [2] 中提到：

R5408M 的四个 2.5G 口是直通 CPU 的，…CPU EN7562C 里面集成有一个千兆交换机，5-8 号千兆口就是走这个交换芯片。…

以及原厂编译配置文件中的

结合 [3] 、[4] 的信息以及使用 ifconfig/iwconfig 命令查看到的网口信息

可以确定，这确实是有一个千兆交换机内置在 en7562ct SoC 之中。GMAC1 扩展支持 SGMII 和 HSGMII

那么，它可能与交换机芯片 EN8855 的驱动是兼容的，在小米AX3000T上也使用了该芯片（很怪，这个芯片应该是达发的，但是被归类到了 MTK，我其实不太确定这个 en 8855 是否能对应上，原机的编译配置使用的驱动是 en8850，因为某些地址可能不对，这些在驱动中有定义）

EN8850 SERIES

EN8850 是达发的一款交换机芯片，支持多种网络接口和协议

下载

估计应该和 EN8850DRE 差不多

或许可以参照 mt7981b-xiaomi-mi-router-common.dtsi 对交换机进行配置。

而设备树中包含了 wdma 和 wed，这两个是达发的硬件加速模块，设备树中使用 en751221 作为兼容硬件，可能同样兼容 EN7526。

Pcie 方面，en7562ct 使用了兼容 mt7520 的 pcie 通道

而 hsgmii 是提供一个 2.5Gbps 内部接口(Interface)，或为 pon/xpon 提供支持。

千兆网口(PHY)来自内部的以太网交换机芯片。

综合以上信息，可以得出结论，en7562ct 芯片是一个多功能的 SoC，将 en7523 作为基础的 CPU，集成了多个功能模块，包括以太网交换机、NPU、PCIe、USB 等。

既 en7523(CPU) + en8850(交换机) 组成了一个 SoC 方案。目前该想法被推翻，详细见后文

如果能够看到 en7523、en7562、7581、en8850、en8855 的 dieshot，或许能更清楚地了解它们的内部结构，以及分析他们的关系。

可以以 an7581 的 dtsi 为基础，修改对应的 CPU 和 NPU 相关信息，以及 GPIO/UART 等，不同设备修改至对应 en7523 的内存位置和范围，添加以太网交换机的配置，就能为作为路由器设备、交换机设备提供 OpenWrt 支持。（至于光猫设备，缺少了 pon/xpon 相关的设备树配置）

额外发现，airoha 的交换机芯片，后缀不一样，它们的结构可能也不相同，故此驱动可能也并不通用，特定型号只能期待 MTK/AIROHA 开源了。

以及可以发现，小米 BE5000，使用了 en7563PT 处理器，推测该芯片应该是 en7562 的更新换代，小米 BE5000 的原版系统是基于 OpenWrt 的，所以还有另一个思路，提取该机型的相关设备树文件，作为 en7562ct 的参考。以及还有一款 极客AP 也使用了 en7563pt 方案。

解包小米固件后，发现该 dts 与 an7562 的内核提取 dts 非常相似，也是基于 an7523 的设备树，但是多了 an7581 的 oop-table 频率表，GPIO、分区表、thermal-zones等，这一差异大概是 airoha SDK 与 OpenWrt 的差异导致的。

另外，我们也可以查看另一个不同架构 SoC en751221 的相关资料，已经有人发表相关的文章。具体见参考资料 [1:1]。也已经有人为其适配 OpenWrt。

en7561 一般搭配 MT7905DEN+MT7975DN 配置成 ax1800
en7562 一般搭配 MT7976DN+MT7916AN 配置成 ax3000
an7963 一般搭配 MT7976CN+MT7991AV 配置成 be5000

OpenWrt 支持的可能性

6月13日记录

今天又有一个新的发现：airoha_ml

随后，我又从这些文件中发现了一个名为 IOWRT Airoha integration，目的是移植 Airoha SDK 到 IOWrt(Based On Openwrt)。

这里面竟然有完整带 eth 的 en7523 的设备树文件，包含了交换机、以太网、NPU、USB 等等。还有驱动之类的东西，XPON/PON 相关的也有。

注意：这个带有以太网、交换机的 en7523.dts 文件可能是错误的说法，因为没有实例可以验证，并且该例子为孤例，没有其他人引用。

所以可以推翻前文的 以 an7581 的 dtsi 为基础 的说法，直接使用 en7523 的 dts 文件进行修改适配就可以了。

不过这个 dts 文件貌似没有使用 SoC 的写法。以及我很好奇，从 dtsi 设备树来看，en7623 应该内置了交换机，为何还要在加上一个 en8850 交换机芯片？难道是因为运营商定制使其成为光猫芯片吗？

所以现在工作量更少了，只需要解决了交换机 en8850 的驱动问题，就可以为该设备提供 OpenWrt 支持。而该驱动可以参考 en8855。

但是又因为 en7523 内置了交换机芯片，或许可以无视该芯片，直接使用 en7523 的设备树进行编译。

以上均为猜想，一切有待验证。

6月15日记录

今天我又发现了一款使用 en7562ct 的设备（不过无线使用 7915）的 make config 文件，发现这是使用21年11月的编译配置，此时它直接使用了 en7523 相关配置，全然没有使用 8850 交换机选项和其他有区别于 en7523 的配置。

而这些变化发生于 22 年某个时间，发生这些变化的 SDK 来自 23 年，所以我更能确定了，早期的 en7562 设备直接使用 en7523 的设备树进行编译。

造成以上迷惑的主要原因来自 OpenWrt 主线的 an7523.dtsi 文件，它太过简陋，缺少了太多相关设备的定义，导致我误以为 en7523 是仅仅是一个 CPU + NPU 的集合体，实际上它也是一个 SoC，前文提到的 en7523(CPU) + en8850(交换机) 的说法是错误的。很可能只是 en7562 是 en7523 的一个变种，增加了 en8850 交换机的某种特性，而这种特性或许并不重要，开启与否可能只是细微的性能差别。

关于交换机，在设备树中的 Frame_Engine 设备中，就有一个交换机的节点。起初查看 /proc/device-tree/ 目录下的设备树文件时，没有发现交换机设备/以太网设备，我还以为是 airoha SDK 的原因，认为实际存在但没在设备树中显示，因为这些设备树的命名难以猜测，以及该设备运行的系统与寻常 linux 不同，似乎叫 tclinux。

并且我查看了 IOWrt Airoha integration 的代码，发现最早他们适配的就是 econet en7562 的设备树，随后改成了 airoha，再后来就是 en75xx 设备树，直到现在的 en7523。甚至他们的 readme 文件举例就是使用 en7562 这颗芯片。

那么，我们可以认为，en7562ct 是一个基于 en7523 的 SoC，可能只是相关功能模块的改进，本质上并无太大区别。

即便不启用属于 en7562 家族的特性，它照样能够运行。

而到如今，IOWrt Airoha integration 项目已经很完善了，未来的某一天应该能看到 OpenWrt 支持。

目前，应该可以尝试使用 IOWrt Airoha integration 的代码进行编译，应该可以启动。

是否好奇没有分区表要怎么启动？我也很好奇，然后查阅了文档，像一些 gpio，分区表，都可以写在 uboot 的环境变量/启动命令中。所以不必纠结分区表了和设备树了。这倒是提醒我备份 u-boot 环境变量十分重要。

此外，我发现，userfs/ 目录下，存在：7523duled.conf、7523guled.conf、7529_62led.conf、7529_cd.conf、led.conf、led_fpga.conf 等文件，这些文件可能是用于配置 LED 灯的。所以，达发的 led gpio 可能是通用的，但也可能是厂商内部定义的。

通过查看 IOWrt Airoha integration 的代码，发现他们的也使用类似 airoha 的 led 配置方式。同样存在于 userfs/led.conf 文件中。

通过比较，可以确定 led.conf 并非是通用的，而是特定于某个设备的配置文件。适配系统时，需要通过 patch 的方式来修改该文件。

6月16日更新

看来为路由器制作系统的想法还是有一些问题，当我下克隆源码后，阅读 README.md 文件时，发现了一段隐藏的文本，我已将其翻译为中文：

这段文本认为只有 USB 以太网加密狗才能作为路由器使用，这让我感到困惑。

这台设备并没有 USB 接口，只有一个串口。难道该设备只能作为交换机使用吗？或者说它的设计初衷就是作为一个交换机？

这也解释了为什么它每次拨号后都无法正常工作的原因了（我在系统中看到它的拨号方法为 PPPoE simulate，实际上上网配置页面并没有 PPPoE 拨号，而是只有通过 IP 上网或 PPP）。

但也可能是因为它的固件有问题，该设备系统 SDK 版本中使用其他方法实现拨号或是存在 BUG。

该项目起始与3年前，他们开发时使用的最初 airoha SDK 可能是 20 年或 21 年的版本，或是 SDK 不完善或是移植过程中的错误，目前查看提交记录，他们似乎跟进了最新的 airoha SDK，版本24Q4，所以这种可能性也可以排除。可以初步确认这是该 SoC 的某种特性或是就是设计了加密。目前查看该项目代码，似乎移植工作基本完成，近期的更新都是小修小补。

不过这也很好了，因为我并不期待这台30块的设备能够胜任主路由器的工作。

另外在前文中提到了 reservearea 分区存储了 mac 地址等信息，还有无线校准信息，另外还一些部分意义不明的区域，可能包含了以太网交换机校准的信息。但是我有一个猜想，那么是否存在一种可能，加密狗信息保存在了 reservearea 分区中？然后通过某种软件的方式实现了路由器功能的解锁？而这种方法受到法律保护？移植 SDK 时不能启用？我不明白。

如果 SoC 内置了 efuse 的话，那么如果要适配系统要非常谨慎了。

但是我也不想去折腾了，如果它仍然只能当 AP/SWITCH 使用，那么目前并不值得去刷系统。如果是玩机的话那就随意了，注意编程器备份 SPI。

个人力量有限，而 IOSysWrt 项目的开发者们已经为达发芯片工作了近3年了，已经接近完善了。想要尝试的话可以使用他们的代码进行编译，或是再次移植到 OpenWrt 上。

目前把该设备当作交换机 + AP 使用，或许是最好的选择。也期待未来这个问题能够被解决。

这 econet 到底是 MTK 的子公司，其芯片（en7562）构成充满了 MTK 技术的影子；SoC 与 mt7622 相似、DMA 可能来自 RaLink、MT7622 的 PCIe、mt7621 的 i2c、mt7530 类似的交换机，结合了 MTK ATF 与 tc3262 的 U-Boot。。。

如果能够完善源码，使得它能正常路由，它将取代 mt7981，毕竟它拥有一颗 NPU，且 RFB 普遍使用 512MB 内存 + 256MB SPI，而且根据特性，对比 mt7981 的 1.3G 频率，它也是能超频的，因为作为 en7523 的升级的 an7581（CPU 部分），它的频率范围在 0.5~1.4GHZ；EN7562CT 的路由器二手价30左右，搭配螃蟹 2.5G 芯片的版本全新价也才 80 左右（截止25年6月）。
